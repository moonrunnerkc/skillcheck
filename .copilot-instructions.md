## Project Identity

**skillcheck** is a cross-agent skill quality gate for `SKILL.md` files conforming to the [agentskills.io specification](https://agentskills.io/specification).

It is not just a linter. It enforces the agentskills.io spec, scores description discoverability, validates file references, and warns about cross-platform compatibility across Claude Code, VS Code/Copilot, OpenAI Codex, Cursor, and other agents.

### Current State (v0.1.0)

The tool currently implements:

- **Frontmatter validation**: required fields (`name`, `description`), character constraints, length limits, reserved-word checks, first/second-person voice detection, XML tag rejection, unknown field warnings.
- **Body sizing**: line-count and token-estimate warnings (heuristic fallback or tiktoken).
- **CLI**: text and JSON output, `--ignore`, `--max-lines`, `--max-tokens`, `--no-color`, exit codes 0/1/2.
- **Parser**: YAML frontmatter extraction, body separation, BOM handling.
- **Architecture**: `parser.py` -> `core.py` (orchestrator) -> `rules/` (individual rule functions) -> `result.py` (diagnostics).

### What Needs to Be Built

The following features close real gaps in the agentskills.io ecosystem. No existing tool covers them.

---

## Feature 1: Full Spec Compliance for the `name` Field

**Gap**: The current `check_name_charset` rule validates lowercase + digits + hyphens but does not enforce all constraints from the agentskills.io spec.

**Required additions** (all in `rules/frontmatter.py`):

1. **Leading/trailing hyphen check** -- names like `-my-skill` or `my-skill-` are invalid per spec.
2. **Consecutive hyphen check** -- names like `my--skill` are invalid per spec.
3. **Directory-name matching** -- VS Code requires the `name` field to match the parent directory name exactly, or the skill silently fails to load. This is a hard failure in production. Rule ID: `frontmatter.name.directory-mismatch`. Severity: ERROR. Compare `skill.frontmatter["name"]` against `skill.path.parent.name`. Add a CLI flag (e.g., `--skip-dirname-check`) for cases where the file is not in its final directory (e.g., CI temp paths).

**Sources**:
- https://agentskills.io/specification (name field constraints)
- https://code.visualstudio.com/docs/copilot/customization/agent-skills (directory-name requirement)
- https://deepwiki.com/agentskills/agentskills/2-agent-skills-format (name constraint details)

## Feature 2: Description Quality Scoring

**Gap**: The description field is the routing mechanism. Agents read descriptions to decide whether to activate a skill. No existing tool scores description quality or helps optimize it for discoverability. `skills-ref` checks existence. `cclint` checks schema. Nobody tells you "this description won't trigger because it lacks action verbs and use-case keywords."

**Implementation** (new module: `rules/description.py`):

Score descriptions 0-100 across these dimensions:

1. **Action verb presence** -- Description should start with or contain imperative/third-person verbs (e.g., "Generates", "Analyzes", "Validates", "Deploys"). Descriptions starting with nouns or articles score lower.
2. **Trigger phrase detection** -- Look for "Use when...", "Activate for...", "Run this skill when...", "Invoke when..." patterns. The Anthropic skill-creator skill itself recommends "pushy" trigger phrases: "Make sure to use this skill whenever the user mentions dashboards, data visualization, internal metrics." Descriptions without trigger context score lower.
3. **Keyword density** -- Measure the ratio of domain-specific, task-identifying keywords vs. generic filler words. "Helps with things" scores low. "Generates conventional commit messages from staged git diffs" scores high.
4. **Specificity** -- Flag vague descriptions. Penalize overuse of generic terms ("tool", "helper", "utility", "stuff", "things") without concrete qualifiers.
5. **Length adequacy** -- Too short (under 20 chars) means no routing signal. Too long (over 500 chars) means the description is doing the job of the body. The sweet spot is descriptive but concise.

**Output**: A `description.quality-score` INFO diagnostic with the numeric score and specific improvement suggestions. Optionally a `--min-desc-score N` CLI flag that promotes scores below threshold to WARNING or ERROR.

**Sources**:
- https://agentskills.io/specification (description field purpose)
- https://deepwiki.com/agentskills/agentskills/2-agent-skills-format (description as routing mechanism)
- https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md (Anthropic's description guidance)

## Feature 3: File Reference Validation

**Gap**: The spec says "use relative paths from the skill root" and "keep file references one level deep from SKILL.md." No tool checks that referenced files actually exist or that reference depth stays at one level.

**Implementation** (new module: `rules/references.py`):

1. **Parse markdown links** -- Extract `[text](path)` and `![alt](path)` references from the body. Filter out URLs (http/https).
2. **Parse code block references** -- Detect `source:` or `file:` directives, `include` patterns, or explicit file paths in fenced code blocks.
3. **Existence check** -- Resolve each relative path against `skill.path.parent` and verify the target exists on disk. Rule ID: `references.broken-link`. Severity: ERROR.
4. **Depth check** -- Flag references deeper than one level from SKILL.md (e.g., `../../other/file.py` or `deeply/nested/path/file.md`). Rule ID: `references.depth-exceeded`. Severity: WARNING.

**CLI integration**: Add `--skip-ref-check` for environments where referenced files are not available (e.g., CI without full checkout).

**Sources**:
- https://agentskills.io/specification (file reference guidance)

## Feature 4: Progressive Disclosure Budget Validation

**Gap**: The agentskills.io spec recommends progressive disclosure across three tiers: metadata (~100 tokens), instructions/body (<5000 tokens), and resources (loaded on demand). No tool validates the token budget across all three tiers.

**Implementation** (extend `rules/sizing.py` or new module `rules/disclosure.py`):

1. **Metadata tier** -- Tokenize frontmatter YAML only. Warn if it exceeds ~100 tokens. Rule ID: `disclosure.metadata-budget`. Severity: WARNING.
2. **Body tier** -- Tokenize the body (excluding frontmatter). Warn if it exceeds 5000 tokens (the spec recommendation, distinct from the existing 8000-token total check). Rule ID: `disclosure.body-budget`. Severity: WARNING.
3. **Bloat detection** -- Flag patterns in the body that belong in referenced files: large code blocks (>50 lines), embedded data tables (>20 rows), base64 content. Rule ID: `disclosure.body-bloat`. Severity: INFO. Suggest moving content to a `resources/` directory.

**Sources**:
- https://agentskills.io/specification (progressive disclosure recommendation: metadata ~100 tokens, instructions <5000 tokens, resources on demand)

## Feature 5: Cross-Agent Compatibility Warnings

**Gap**: The agentskills.io spec is now consumed by Claude Code, VS Code/Copilot, OpenAI Codex, Cursor, and others. Implementation-specific fields work in one agent but are ignored or cause breakage in others. No validator flags this.

**Implementation** (new module: `rules/compat.py`):

Define a compatibility matrix:

| Field | Claude Code | VS Code/Copilot | Codex | Cursor |
|---|---|---|---|---|
| `name` | supported | supported (must match dirname) | supported | supported |
| `description` | supported | supported | supported | supported |
| `model` | supported | ignored | unknown | unknown |
| `disable-model-invocation` | supported | ignored | unknown | unknown |
| `mode` | supported | ignored | unknown | unknown |
| `allowed-tools` | supported | supported | unknown | unknown |
| `user-invocable` | supported | supported | unknown | unknown |
| `hooks` | supported | ignored | unknown | unknown |
| `context` | supported | supported | unknown | unknown |

Rules to implement:

1. **Claude-only field warning** -- Flag fields like `model`, `disable-model-invocation`, `mode`, `hooks` with: "This field is Claude Code-specific and will be ignored by VS Code/Copilot and other agents." Rule ID: `compat.claude-only`. Severity: INFO.
2. **VS Code dirname requirement** -- Already covered in Feature 1, but add an INFO note: "VS Code requires name to match the parent directory. This file would fail to load in VS Code." Rule ID: `compat.vscode-dirname`. Severity: INFO (or ERROR if `--strict-vscode` flag is set).
3. **Unknown agent behavior** -- For fields where behavior in Codex/Cursor is unknown, note: "Behavior of field 'X' in Codex and Cursor is unverified." Rule ID: `compat.unverified`. Severity: INFO.

**CLI integration**: Add `--target-agent claude|vscode|all` flag (default: `all`) to scope compatibility checks. When targeting a specific agent, promote relevant warnings to errors.

**Sources**:
- https://code.visualstudio.com/docs/copilot/customization/agent-skills (VS Code behavior)
- https://leehanchung.github.io/blogs/2025/10/26/claude-skills-deep-dive/ (Claude Code-specific fields)

## Architecture Notes

### File Layout (target state)

```
src/skillcheck/
    __init__.py
    cli.py              # Argument parsing, output formatting, entry point
    config.py           # Constants, thresholds, known fields, compat matrix
    core.py             # Orchestrator: parse -> rules -> filter -> result
    parser.py           # YAML frontmatter extraction, body separation
    result.py           # Diagnostic, Severity, ValidationResult dataclasses
    tokenizer.py        # Token estimation (tiktoken or heuristic fallback)
    rules/
        __init__.py     # get_rules() aggregator
        frontmatter.py  # Name, description, field validation
        sizing.py       # Line-count, token-count thresholds
        description.py  # NEW: Description quality scoring (0-100)
        references.py   # NEW: File reference validation
        disclosure.py   # NEW: Progressive disclosure budget checks
        compat.py       # NEW: Cross-agent compatibility warnings
        patterns.py     # Reserved for anti-pattern detection (stub)
```

### Rule Function Contract

Every rule is a callable with signature:

```python
def check_something(skill: ParsedSkill) -> list[Diagnostic]:
```

Rules return an empty list when no issues are found. Rules never raise exceptions for validation failures; they return Diagnostic objects. Rules may access `skill.path`, `skill.frontmatter`, `skill.body`, `skill.body_lines`, and `skill.raw_text`.

### Diagnostic Severity Usage

- **ERROR**: Spec violations that will cause a skill to fail to load, be rejected by agents, or behave incorrectly in production.
- **WARNING**: Content that technically loads but violates recommendations, exceeds budgets, or degrades quality.
- **INFO**: Suggestions, quality scores, compatibility notes, and optimization guidance.

### CLI Exit Codes

- `0`: No errors (warnings and info are allowed).
- `1`: One or more errors found.
- `2`: Input error (missing file, empty directory, invalid arguments).

### Adding New CLI Flags (anticipated)

| Flag | Feature | Purpose |
|---|---|---|

### Testing Strategy

- Every new rule gets unit tests with at least: one passing case, one failing case, and one edge case.
- Fixture files go in `tests/fixtures/` with descriptive names (e.g., `bad_name_leading_hyphen.md`, `valid_with_refs.md`).
- Integration tests validate full CLI output for multi-rule interactions.
- Quality scoring tests use known-good and known-bad descriptions with expected score ranges (not exact values, to allow tuning).
- File reference tests use a temporary directory structure with real files and broken links.

### Dependencies

- Runtime: `pyyaml>=6.0` (required), `tiktoken>=0.7` (optional, for accurate token counts).
- Dev: `pytest>=8.0`.
- No new runtime dependencies should be added without justification. The tool must stay lightweight and fast.
